# 02 — Architecture Overview

## Overall Architecture Style

This repository uses a **Layered Infrastructure Architecture** — three Terraform layers that build on each other — combined with an **Azure Functions event-driven automation layer** for post-deployment provisioning tasks.

There is no single monolithic architecture style. Each layer has its own purpose and dependencies flow in one direction: base → customer ← functions.

---

## The Three-Layer Terraform Model

```
┌─────────────────────────────────────────────────────────────┐
│  LAYER 3: sas-sql-storage/                                  │
│  Purpose: Customer blob storage + SAS token creation        │
│  Reads: base outputs via terraform_remote_state             │
└─────────────────────────────────────────────────────────────┘
                         ▲
┌─────────────────────────────────────────────────────────────┐
│  LAYER 2: customer/                                         │
│  Purpose: Per-firm isolated resources                       │
│  Reads: base outputs via terraform_remote_state             │
│  Config: config/{env}/{firmname}.tfvars.json                │
└─────────────────────────────────────────────────────────────┘
                         ▲
┌─────────────────────────────────────────────────────────────┐
│  LAYER 1: base/                                             │
│  Purpose: Shared infrastructure, deployed once per env      │
│  Config: base/{nonprod,prod}.tfvars.json                   │
└─────────────────────────────────────────────────────────────┘
```

**Why this layering?**

- The base layer is expensive and slow to create — it only needs to exist once per environment.
- Isolating customer state means a broken customer deployment cannot affect other customers.
- The `terraform_remote_state` data source lets the customer layer read base outputs (Key Vault URI, SQL server FQDN, App Service Plan ID, etc.) without duplicating values.

### How Remote State Works

The customer layer's `data.tf` contains:
```hcl
data "terraform_remote_state" "base" {
  backend = "azurerm"
  config = {
    storage_account_name = var.backendStorageAccount
    container_name       = "tfstate"
    key                  = "base-${var.environment}.tfstate"
    access_key           = var.TFSTATE_SECRET
  }
}
```

This gives the customer layer access to `data.terraform_remote_state.base.outputs.*` — any value that `base/output.tf` exports. Examples: SQL server FQDN, Key Vault URI, App Service Plan IDs.

> **Warning:** The exact configuration in `customer/data.tf` is authoritative. If you need to reference it, always read that file — do not copy simplified examples.

## Layer 1: Base Infrastructure

**Deployed:** Once per environment (nonprod, prod).

**Key resources created:**
- **VNet + Subnets** — Private network for all shared resources
- **Application Gateway (WAF)** — Single HTTPS entry point for all customer web apps; performs SSL termination and WAF inspection
- **Azure Key Vault** — Stores all secrets (B2C credentials, SQL passwords, app secrets)
- **SQL Server** — Shared Azure SQL logical server with elastic pools
- **CosmosDB** — NoSQL store for O365 tenant application metadata
- **Service Bus** — Async message queue for inter-service communication
- **Log Analytics Workspace** — Central destination for all diagnostic logs
- **Storage Account** — Blob storage for Terraform state and application use
- **Azure Functions App** — Hosts the provisioning automation functions
- **Logic App** — Workflow automation for post-deployment tasks
- **API Web App** — Internal API service

**Naming pattern:**
```
{org_id}-{region}-{environment}-{function}-{index}
# e.g., adv-uks-nonprod-lc-001
```

All resource names are generated by the `azurecaf_name` resource via the `caf_name` module to guarantee compliance with Azure naming rules and the CAF standard.

---

## Layer 2: Customer Resources

**Deployed:** Once per customer per environment.

**Key resources created per customer:**
- **Windows App Service** — The customer's PCMS web application
- **SQL Database** — Isolated database on the shared SQL server's elastic pool
- **Application Insights** — Customer-specific telemetry
- **B2C App Registration** — Azure AD B2C app for authentication
- **B2C Custom Policy** — Uploaded trust framework XML
- **App Gateway routing** — Backend pool + listener + rules for the customer hostname
- **Deployment Scripts** — Run via Azure Container Instances to execute B2C setup Bash scripts

**Naming pattern:**
```
{org_id}-{region}-{environment}-{function}-{customer_name}
# e.g., adv-uks-nonprod-lc-contoso
```

**Customer config file shape** (`config/nonprod/contoso.tfvars.json`):
```json
{
  "customer": {
    "name": "contoso",
    "firmDomain": "contoso.onmicrosoft.com",
    "dbSku": "GP_Gen5",
    "elasticPoolIndex": 1,
    "appServicePlanIndex": 1,
    ...
  },
  "bacpacUrl": "https://storage.blob.core.windows.net/...",
  "sqlAdministratorLogin": "...",
  "sqlAdministratorLoginPassword": "..."
}
```

## Layer 3: SAS SQL Storage

**Deployed:** On demand, for specific customers that need time-limited blob access.

Creates a blob container and generates a SAS token with read/write/list permissions, stores the token in Key Vault. Some workflows use this instead of sharing permanent storage credentials.

## Azure Functions Layer (Post-Deployment Automation)

The `functions/` directory contains C# .NET 8 Azure Functions that run **after** Terraform provisions infrastructure. They handle tasks that cannot be expressed in Terraform:

| Function | Trigger | What It Does |
|---|---|---|
| `DatabaseUpgrade` | HTTP | Runs SQL schema migrations against a customer DB |
| `LongRunningDbUpdate` | HTTP (Durable) | Long-running DB upgrades via Durable Functions orchestration |
| `UdfDatabaseUpgrade` | HTTP | UDF (User-Defined Function) schema migrations |
| `OneDriveIntegration` | HTTP | Creates M365 Groups, Teams, SharePoint libraries, OneDrive setup |
| `ByobiSetup` | HTTP | Executes stored procedures for Business Intelligence provisioning |
| `O365TenantAppManagementFunction` | HTTP | CRUD for Cosmos DB tenant application records |
| `SecretRotation` | Timer (every 10 days) | Rotates O365 app secrets via Graph API, syncs to Key Vault |
| `CheckExchangeSync` | HTTP | Checks Exchange integration status, updates Cosmos DB |
| `ThirdFortWebHook` | HTTP | Validates HMAC-SHA256 signatures, posts to Service Bus |

The Function App is deployed to the shared Function App resource created by the base layer.

## Authentication Architecture

Three distinct authentication patterns exist in this codebase:

### 1. Managed Identity (preferred, production)
Azure resources authenticate to each other using `DefaultAzureCredential`. No passwords stored anywhere. Used for:
- Functions → Key Vault
- Functions → CosmosDB
- App Services → Key Vault

### 2. Confidential Client (service principal, multi-tenant)
When a function must act as a service principal in a **customer's** Azure AD tenant (not ours), it uses `ConfidentialClientApplicationBuilder` with a client ID + secret from Key Vault. Used for:
- OneDrive provisioning in customer tenant
- B2C policy deployment

### 3. Interactive Browser (local development only)
When `Environment == "Development"`, `InteractiveBrowserCredential` is used so developers can run functions locally without managed identity.

## Multi-Tenancy Model

This is not a code-level multi-tenancy pattern (no tenant filter in SQL queries). Instead, it is **infrastructure-level isolation**:

- Each customer has their own **SQL Database** (on a shared server)
- Each customer has their own **App Service** (on a shared App Service Plan)
- Each customer has their own **Application Insights** resource
- Connection strings are fetched from **Key Vault** at runtime — each customer's app has its own Key Vault secret

The benefit: a bug or crash in one customer's app cannot access another customer's data. The tradeoff: 270+ resources of each type to manage.

---

## Networking Model

All traffic enters through the **Application Gateway**:
- Terminates TLS
- Routes by hostname (`<firmname>.legalcloud.com`) to the correct App Service backend pool
- Applies WAF (Web Application Firewall) rules

Private resources (SQL, CosmosDB, Storage) are accessed via **Private Endpoints** — they have no public internet exposure. App Services communicate with them over the VNet.

APIM (API Management) has dedicated IP restrictions configured on web apps — only APIM's public and private IPs can access the API layer.

## State Management

| Layer | State File Location |
|---|---|
| base (nonprod) | `tfstate` container, key: `base-nonprod.tfstate` |
| base (prod) | `tfstate` container, key: `base-prod.tfstate` |
| customer (nonprod) | `tfstate` container, key: `<customername>.terraform.tfstate` |
| customer (prod) | `tfstate` container, key: `<customername>.terraform.tfstate` |

State files are in Azure Storage. Access requires the `ARM_ACCESS_KEY` or `TFSTATE_SECRET` environment variable. **Never modify state files manually** — use `terraform state` commands with care.

## CI/CD Architecture (Harness)

Harness pipelines orchestrate the full deployment lifecycle:

```
PR Raised
    │
    ├── buildbasepr.yaml       ← terraform validate + plan (base)
    └── buildcustomerpr.yaml   ← terraform validate + plan (customer)

Merge to main
    │
    ├── deploybasenonprod.yaml ← terraform apply (base, nonprod)
    └── deployapplicationnonprod.yaml ← terraform apply (customer, nonprod)

Manual trigger (customer onboarding)
    │
    └── terraformprovisioncustomer.yaml
            ├── terraform apply (customer layer)
            ├── Invoke DatabaseUpgrade function
            ├── Invoke OneDriveIntegration function
            └── Invoke ByobiSetup function (if requested)
```

Reusable steps live in `.harness/orgs/Legal/projects/PCMS/templates/step/` so they can be shared across pipelines without duplication.

## Key Architectural Constraints You Must Respect

1. **One-way data flow:** Customer layer reads from base via `terraform_remote_state`. It never writes back. Do not try to modify base resources from customer Terraform.

2. **State isolation:** Each customer has its own state file. Running `terraform apply` for customer A cannot affect customer B.

3. **Naming is governed:** Resource names come from `azurecaf_name`. Do not hardcode Azure resource names.

4. **Secrets stay in Key Vault:** No connection strings, passwords, or API keys in Terraform variables, code, or config files. They are injected at runtime from Key Vault.

5. **Base modules are shared:** A change to `base/modules/sql_server/` affects every environment. Test carefully.
