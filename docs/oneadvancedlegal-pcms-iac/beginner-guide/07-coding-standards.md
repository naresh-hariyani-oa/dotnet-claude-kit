# 07 — Coding Standards

## Terraform Standards

### Naming Conventions

All Azure resource names must be generated by the `azurecaf_name` resource. Never hardcode a resource name.

**Correct:**
```hcl
resource "azurecaf_name" "kv" {
  name          = "${var.org_id}-${var.region}-${var.environment}-${var.function}"
  resource_type = "azurerm_key_vault"
}

resource "azurerm_key_vault" "main" {
  name = azurecaf_name.kv.result
  ...
}
```

**Wrong:**
```hcl
resource "azurerm_key_vault" "main" {
  name = "my-keyvault"  ← hardcoded, not CAF-compliant
  ...
}
```

**Naming patterns (from CLAUDE.md):**

| Context | Pattern | Example |
|---|---|---|
| Shared (base) resources | `{org_id}-{region}-{env}-{function}-{index}` | `adv-uks-nonprod-lc-001` |
| Customer resources | `{org_id}-{region}-{env}-{function}-{customer}` | `adv-uks-nonprod-lc-contoso` |
| Function Apps | `fn-{org_id}-{region}-{env}-{function}-{index}` | `fn-adv-uks-nonprod-lc-001` |
| Logic Apps | `{org_id}-{region}-{env}-la-{index}` | `adv-uks-nonprod-la-001` |
| Deployment Scripts | `ds-{org_id}-{region}-{env}-{function}-{customer}-{purpose}` | |

Common values: `org_id = "adv"`, `region = "uks"`, `function = "lc"`

---

### Tagging

All Azure resources must include standard tags. Check existing resources for the required tag set — it includes at minimum:
- `environment`
- `application`
- Cost centre tag

Pass tags as a `tags` variable or local and apply to every resource block.

---

### Variables vs Locals

| Use case | Use |
|---|---|
| Value passed in from outside (tfvars) | `variable` |
| Computed or derived value | `local` |
| Value referenced from another layer | `data "terraform_remote_state"` output |
| Sensitive value (connection strings, passwords) | Key Vault — never in variables |

Do not hardcode in `main.tf` what should be a variable. Do not put computed logic in `variables.tf` — that's what `locals.tf` is for.

---

### Secrets

**Never** put passwords, connection strings, or API keys in:
- `*.tfvars.json` files (they are in Git)
- `variables.tf` default values
- Terraform outputs
- Code files

Connection strings are constructed in `locals.tf` and reference values retrieved at runtime from Key Vault. Secrets are stored in Key Vault by the base layer and consumed by the customer layer at deploy time.

---

### Module Conventions

Each module must have:
- `main.tf` — resource definitions
- `variables.tf` — all inputs declared with `type` and `description`
- `output.tf` — export values that callers need

Modules should not reference `var.environment` directly unless it's in their own `variables.tf`. Pass everything in explicitly.

Do not add resources to a module that don't belong to that module's concern. The `key_vault` module creates Key Vault resources; it doesn't also create storage accounts.

---

### File Organisation

```
base/main.tf         ← Resource definitions grouped by service area
base/locals.tf       ← All computed locals
base/variables.tf    ← All input variables with descriptions
base/output.tf       ← All outputs
base/network.tf      ← Networking resources (VNet, subnets, NSGs)
base/agw.tf          ← Application Gateway resources
```

Split large `main.tf` files by resource domain (network, security, compute). Do not put everything in one file if it exceeds ~200 lines.

## C# / Azure Functions Standards

### Naming

| Element | Convention | Example |
|---|---|---|
| Classes | PascalCase | `DatabaseOperations`, `TenantDBProvider` |
| Methods | PascalCase | `GetAllTenantApplicationsAsync` |
| Private fields | `_camelCase` | `_httpClient`, `_logger` |
| Local variables | camelCase | `connectionString`, `firmId` |
| Constants | PascalCase or UPPER_SNAKE | `RetryDelaySeconds`, `FunctionConstants.KvEndpointEnvVar` |
| Async methods | Must end with `Async` | `GetSecretAsync`, `UpsertTenantApplicationAsync` |

---

### Async/Await

All I/O operations must be async. Never use `.Result` or `.Wait()` on a Task — this blocks the thread and can cause deadlocks.

**Correct:**
```csharp
var secret = await KeyVaultManager.GetSecretAsync(kvEndpoint, secretName);
```

**Wrong:**
```csharp
var secret = KeyVaultManager.GetSecretAsync(kvEndpoint, secretName).Result;  ← blocks thread
```

Make the calling method `async Task` or `async Task<T>`. Propagate `async` up the call chain.

---

### Logging

Use `ILogger` (injected via `ILoggerFactory`) with structured logging parameters:

```csharp
// Correct — structured, searchable in App Insights
_logger.LogInformation("Starting database upgrade for firm {FirmId}", firm.Id);
_logger.LogError(ex, "Database upgrade failed for firm {FirmId} at version {Version}", firmId, version);

// Wrong — string interpolation, not searchable
_logger.LogInformation($"Starting database upgrade for {firm.Id}");
```

Use appropriate log levels:
- `LogDebug` — verbose, only useful for deep diagnostics
- `LogInformation` — normal operation events
- `LogWarning` — unexpected but recoverable situations
- `LogError` — failures that affect a single operation
- `LogCritical` — failures that affect the entire service

---

### SQL Access

Use parameterized queries or stored procedures. Never build SQL strings via string interpolation or concatenation.

**Correct:**
```csharp
var cmd = new SqlCommand("SELECT * FROM SysParam WHERE name = @name", connection);
cmd.Parameters.AddWithValue("@name", paramName);

// Or via stored procedure:
var cmd = new SqlCommand("[support].[usp_BYOBI_InitialSetup]", connection);
cmd.CommandType = CommandType.StoredProcedure;
cmd.Parameters.AddWithValue("@Tables", tableList);
```

**Wrong:**
```csharp
var cmd = new SqlCommand($"SELECT * FROM SysParam WHERE name = '{paramName}'", conn);
// ↑ SQL injection vulnerability
```

---

### Error Handling

- **Do not swallow exceptions** without logging them.
- **Do not remove existing error handling** in PR reviews without strong justification.
- Catch specific exception types rather than bare `Exception` where possible.
- For Key Vault 404 (secret not found), this is an expected case — catch it explicitly:

```csharp
catch (Azure.RequestFailedException ex) when (ex.Status == 404)
{
    // Secret doesn't exist yet — handle accordingly
}
```

- Functions that process multiple items (like `SecretRotation` iterating tenants) should catch per-item, log the error, and continue — not abort the entire run.

---

### Dependency Injection

The Functions host is configured in `Program.cs`. Currently, minimal services are registered:

```csharp
services.AddApplicationInsightsTelemetryWorkerService();
services.ConfigureFunctionsApplicationInsights();
```

Functions receive `ILoggerFactory` via constructor injection. Do not use `ILogger<T>` directly as a constructor parameter (it is not registered in the isolated worker model by default) — create from `ILoggerFactory`:

```csharp
public MyFunction(ILoggerFactory loggerFactory)
{
    _logger = loggerFactory.CreateLogger<MyFunction>();
}
```

If you add new registered services, add them in `Program.cs`. Do not use `new()` for services that should be injected.

---

### Configuration Access

Read configuration from environment variables using `Environment.GetEnvironmentVariable`:

```csharp
var kvEndpoint = Environment.GetEnvironmentVariable("AZURE_KEY_VAULT_ENDPOINT");
```

For sensitive values, fetch from Key Vault using `KeyVaultManager.GetSecretAsync`. Do not cache Key Vault secrets indefinitely — the rotation function exists to update them.

---

### CosmosDB Access Conventions

The `TenantDBProvider` uses the CosmosDB SDK's `PatchOperation` for partial updates — this avoids overwriting fields you did not intend to change:

```csharp
// Correct — only patch non-null fields
var patches = new List<PatchOperation>();
if (updatedApp.Secret != null)
    patches.Add(PatchOperation.Set("/secret", updatedApp.Secret));
await container.PatchItemAsync<O365TenantApplications>(id, partitionKey, patches);
```

Do not read-then-overwrite full documents when you only need to update one or two fields.

### File-Level Organisation (Functions)

Follow the existing folder structure:

```
Functions/                  ← Function classes (one class per function domain)
Functions/OneDriveIntegration/
  Managers/                 ← Business logic managers (don't do I/O in managers)
  Models/                   ← Plain data models (no logic)
  Persistance/              ← Database context classes (connection management)
  Providers/                ← Data access layer (queries and writes)
Functions/Common/           ← Constants and enums for the Functions project
Legal.Cloud.Provisioning.Common/
  Managers/                 ← Azure SDK client wrappers (auth, Key Vault, Graph)
  Models/                   ← Shared domain models
  Providers/                ← Shared providers (e.g., TokenProvider)
```

Note: the folder is spelled `Persistance` (not `Persistence`) — follow existing spelling to avoid confusion.

---

### Function Trigger Conventions

- **HTTP functions:** Use `AuthorizationLevel.Function` for external-facing endpoints (requires function key). `AuthorizationLevel.Anonymous` is only appropriate for functions behind the App Gateway/APIM with their own auth.
- **Timer functions:** Document the cron expression with a comment explaining the schedule in plain English.
- **Service Bus functions:** Use `maxConcurrentCalls: 1` unless parallel processing is explicitly designed for (see `host.json`).

---

## General Conventions

- **No comments for obvious code.** Only add a comment when the WHY is non-obvious — a hidden constraint, a workaround, or a surprising behaviour.
- **No TODO comments in committed code.** Raise a Jira/GitHub issue instead.
- **No hardcoded values** — use constants from `FunctionConstants.cs` or `ApplicationConstants.cs`.
- **PR descriptions** must follow the template in `.github/pull_request_template.md` — include reason, changes, impact, risk level, and testing.
