# 03 — Design Patterns

This document describes the actual design patterns found in this codebase, where they are implemented, and how to work with them correctly.

---

## Terraform Side

### Pattern: Module Composition

**Where:** `base/main.tf`, `customer/main.tf`

**What it is:** Rather than writing raw resource blocks everywhere, related resources are grouped into reusable modules in `base/modules/` and `customer/modules/`. The root module calls child modules via `module "name" { source = "./modules/..." }`.

**Why:** Modules enforce consistency. The `caf_name` module ensures every resource uses the same naming logic. The `sql_server` module ensures every SQL deployment gets auditing, AD admin, and diagnostic settings — you can't forget them because they're baked in.

**How to extend safely:**
- If you need a new resource of a type that already has a module, call that module. Don't write a raw resource block that bypasses it.
- If you need a new module, follow the same shape as existing ones: `main.tf`, `variables.tf`, `output.tf`, and always include a `caf_name` sub-module call for the resource name.

**Example call:**
```hcl
module "key_vault" {
  source         = "./modules/key_vault"
  name           = module.caf_name.result
  resource_group = azurerm_resource_group.main.name
  location       = var.location
  tags           = local.tags
}
```

### Pattern: Remote State as Interface

**Where:** `customer/data.tf`, `sas-sql-storage/data.tf`

**What it is:** The base layer publishes outputs (`base/output.tf`). Customer layers consume those outputs via `data "terraform_remote_state" "base"`. This is the only allowed cross-layer communication.

**Why:** It decouples layers. The base layer can be updated independently. Customer layers get stable values without needing to know how base resources are implemented.

**How to extend safely:**
- Need a new value from base in the customer layer? Add it to `base/output.tf` first, then reference it as `data.terraform_remote_state.base.outputs.<new_output>` in the customer layer.
- Never pass base resource IDs as Terraform variables to the customer layer — that's what remote state is for.

### Pattern: CAF Naming Module

**Where:** Used in every module via `module "caf_name"` calling `./modules/caf_name` or directly calling `azurecaf_name` resources.

**What it is:** Every Azure resource name is generated by the `azurecaf_name` resource from the `aztfmod/azurecaf` provider. It enforces Azure naming rules, length limits, and the org naming convention automatically.

**Why:** Azure has strict and inconsistent naming rules per resource type (e.g., Storage accounts: 3-24 lowercase alphanumeric; Key Vaults: 3-24 alphanumeric+hyphens). Doing this manually leads to errors. The `azurecaf_name` resource knows all the rules.

**How to extend safely:**
- When adding a new resource, always use `azurecaf_name` to generate its name.
- Follow the naming convention: `{org_id}-{region}-{environment}-{function}-{customer_or_index}`
- Use the correct `resource_type` string (e.g., `"azurerm_key_vault"`, `"azurerm_app_service"`)

### Pattern: Deployment Script Wrapper (azapi_resource)

**Where:** `customer/modules/azapi_resource/`, `sas-sql-storage/modules/azapi_resource/`

**What it is:** Azure Container Instances–backed deployment scripts, accessed via the `azapi` provider. They run Bash or PowerShell scripts inside Azure during `terraform apply`.

**Why:** Some provisioning steps — like B2C app registration creation or Graph API permission grants — cannot be done with Terraform resource types. The `azapi_resource` module wraps these as infrastructure that runs scripts inline.

**How to extend safely:**
- This module is a thin wrapper. If you add a new deployment script, add it to `customer/scripts/`, then wire it up using an `azapi_resource` module call.
- Be aware these scripts run in Azure context — they have access to the managed identity credentials of the App Service, not your local machine.

## C# / Azure Functions Side

### Pattern: Provider Pattern (Data Access)

**Where:**
- `functions/Legal.Cloud.Provisioning.Functions/OneDriveIntegration/Providers/`
- `functions/Legal.Cloud.Provisioning.Functions/Tenants/Providers/TenantDBProvider.cs`

**What it is:** Providers wrap a database context and expose specific query/write methods. The calling function class receives a provider (or creates one) and calls methods on it rather than writing queries inline.

**Example:**
```
TenantDBContext     ← manages the CosmosDB client connection
    └── TenantDBProvider   ← exposes GetAllTenantApplicationsAsync(), UpsertTenantApplicationAsync()
        └── O365TenantAppManagementFunction  ← calls provider methods
```

**Why:** Separates connection management from query logic, and query logic from business logic. Makes it easier to change how data is stored without rewriting the function.

**How to extend safely:**
- If you need to add a new query, add it to the relevant Provider class.
- Do not write SQL or CosmosDB queries directly inside function classes.
- If the provider class does not exist for your use case, create one following the same pattern: constructor takes a context or connection string, methods are `async Task<T>`.

### Pattern: Manager Pattern (Infrastructure Clients)

**Where:**
- `functions/Legal.Cloud.Provisioning.Common/Managers/KeyVaultManager.cs`
- `functions/Legal.Cloud.Provisioning.Common/Managers/AuthenticationManager.cs`
- `functions/Legal.Cloud.Provisioning.Common/Managers/GraphClientManager.cs`

**What it is:** Manager classes encapsulate the creation and configuration of Azure SDK clients. They are currently static classes with static methods — call them like utilities.

**Example:**
```csharp
// Don't do this in function code:
var client = new SecretClient(new Uri(kvUri), new DefaultAzureCredential());

// Do this instead:
var secret = await KeyVaultManager.GetSecretAsync(kvUri, secretName);
```

**Why:** Centralises credential strategy, retry config, and environment-specific logic (e.g., swapping `DefaultAzureCredential` for `InteractiveBrowserCredential` in dev mode) in one place.

**How to extend safely:**
- When a function needs to call a new Azure service, check if a Manager for it exists first.
- If you're adding a new Manager class, put it in `Legal.Cloud.Provisioning.Common/Managers/` and follow the static method pattern.

### Pattern: Durable Functions Orchestration

**Where:** `functions/Legal.Cloud.Provisioning.Functions/LongRunningDbUpdate.cs`

**What it is:** For operations that can take many minutes, the [Azure Durable Task Framework](https://learn.microsoft.com/en-us/azure/azure-functions/durable/durable-functions-overview) is used. It has three function types:
1. **HTTP Starter** — receives the HTTP request, starts the orchestration, returns `202 Accepted` with a status URL
2. **Orchestrator** — defines the workflow; calls activity functions; implements retry policy
3. **Activity** — does the actual work (runs the DB upgrade); can be retried independently

**Retry policy in this codebase:**
```
Max 3 attempts
First retry: 20 minutes
Max interval: 60 minutes
Backoff coefficient: 2x
```

**Why:** Normal Azure Functions time out after 5–10 minutes. Database upgrades for large firms can take significantly longer. Durable Functions checkpoint their state in storage, so a timeout or restart resumes from where it left off.

**How to extend safely:**
- If you add a new long-running operation, follow this same three-function pattern (starter → orchestrator → activity).
- Do not put the actual work in the orchestrator — only in activity functions. Orchestrators must be deterministic (no I/O, no randomness, no DateTime.Now).
- The Durable Hub name is `DbUpdateAppHub` — all Durable Functions in this app share it.

### Pattern: Configuration via Environment Variables + Key Vault

**Where:** Throughout all function classes.

**What it is:** A two-step configuration pattern:
1. Non-sensitive config comes from environment variables (e.g., `Environment.GetEnvironmentVariable("COSMOS_DB_ENDPOINT")`)
2. Sensitive config (connection strings, secrets) is fetched at runtime from Key Vault using the Key Vault URI from an environment variable

**Why:** This follows [12-factor app principles](https://12factor.net/config). The function app has no hardcoded credentials. In Azure, environment variables are set via App Service application settings. Locally, they go in `local.settings.json` (not checked into git).

**Key environment variables:**
```
AZURE_KEY_VAULT_ENDPOINT          ← URI of the Key Vault
COSMOS_DB_ENDPOINT                ← CosmosDB URI
COSMOS_DB_DATABASE                ← CosmosDB database name
COSMOS_DB_CONTAINER               ← CosmosDB container name
OnboardingServiceBus              ← Service Bus connection string
TEAMS_WEBHOOK_URL                 ← Teams alert webhook
MAINTAINANCE_CLIENT_ID            ← Service principal for Graph API
MAINTAINANCE_CLIENT_SECRET        ← Service principal secret (from KV)
DatabaseUpgradeIncludeFirms       ← Whitelist of firm IDs (optional)
DatabaseUpgradeExcludedFirms      ← Blacklist of firm IDs (optional)
Environment                       ← Set to "Development" for local dev
```

**How to extend safely:**
- Add new environment variables to `local.settings.json` template for local dev.
- Never hardcode URLs, names, or credentials — always read from environment variables.
- For new secrets, store them in Key Vault and read via `KeyVaultManager`.

### Pattern: HMAC Signature Validation (Security Boundary)

**Where:** `functions/Legal.Cloud.Provisioning.Functions/ThirdFortWebHook.cs`

**What it is:** Incoming webhook messages are validated by computing HMAC-SHA256 of the request body with a shared secret and comparing it to the signature in the request header. Uses constant-time comparison to prevent timing attacks.

**Why:** Without this, anyone who discovers the webhook URL could send fake messages. The HMAC proves the sender knows the shared secret.

**How to extend safely:**
- If you add new external webhook integrations, apply the same pattern: validate signature before processing body.
- Do not log the raw request body before validation (it may contain sensitive data).

---

### Pattern: Retry with Exponential Backoff (Graph API)

**Where:** `functions/Legal.Cloud.Provisioning.Functions/SecretRotation.cs`

**What it is:** When calling Microsoft Graph API, the code catches `429 TooManyRequests` and `503 ServiceUnavailable` responses, reads the `RetryAfter` response header (or defaults to 5 seconds), and retries recursively.

**Why:** Graph API is rate-limited. Long-running operations iterating over all tenant applications will eventually hit throttling. Without retry logic, the function fails halfway through.

**How to extend safely:**
- Any new Graph API code that iterates over large sets should include the same retry pattern.
- The `RetryDelaySeconds` constant is 5 — this is the fallback when the API does not specify a retry-after value.

## Summary: Pattern Quick Reference

| Pattern | Location | Purpose |
|---|---|---|
| Module Composition | `base/main.tf`, `customer/main.tf` | Reusable, consistent resource groups |
| Remote State Interface | `customer/data.tf` | Cross-layer value sharing |
| CAF Naming | Every module | Azure-compliant resource names |
| Deployment Script Wrapper | `customer/modules/azapi_resource` | Run scripts during terraform apply |
| Provider Pattern | `*/Providers/*.cs` | Data access abstraction |
| Manager Pattern | `Common/Managers/*.cs` | Azure SDK client encapsulation |
| Durable Orchestration | `LongRunningDbUpdate.cs` | Long-running workflows with retry |
| Env Var + Key Vault Config | All function classes | Zero secrets in code |
| HMAC Validation | `ThirdFortWebHook.cs` | Webhook authenticity verification |
| Graph API Retry Backoff | `SecretRotation.cs` | Resilient API calls |
