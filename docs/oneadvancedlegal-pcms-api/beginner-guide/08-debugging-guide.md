# 08 — Debugging Guide

## Log File Location

Runtime logs are written to:

```
api/IntegrationApi/Logs/LogFile.txt
```

Rolling strategy: daily, max 10 MB per file, 10 files retained. In development the minimum log level is `Debug`, so you will see detailed output including SQL queries (when EF Core SQL logging is enabled).

---

## Log Categories

Serilog maps namespaces to short category labels:

| Category | Namespace | What you see |
|---|---|---|
| `BUS` | `Legal.Pcms.Integration.Api.Business.*` | Service method entry/exit, exceptions |
| `DTA` | `Legal.Pcms.Integration.Api.Data.*` | Repository operations, SQL results |
| `SYS` | `Microsoft.*` | ASP.NET Core framework logs |

To filter logs to a specific domain in development, adjust `appsettings.Development.json`:

```json
"Serilog": {
  "MinimumLevel": {
    "Default": "Warning",
    "Override": {
      "Legal.Pcms.Integration.Api.Business.Matter": "Debug",
      "Legal.Pcms.Integration.Api.Data.Matter": "Debug"
    }
  }
}
```

---

## Enabling EF Core SQL Logging

To see the actual SQL generated by Entity Framework Core:

```json
"Serilog": {
  "MinimumLevel": {
    "Override": {
      "Microsoft.EntityFrameworkCore.Database.Command": "Information"
    }
  }
}
```

This prints every SQL statement and its parameters to the log. Disable in production.

## Correlation IDs

Every request gets a correlation ID. It appears in:
- Log lines for the request
- The `X-Correlation-ID` response header
- The body of 500 error responses

To trace a specific request through logs:

```bash
grep "abc-123-correlation-id" api/IntegrationApi/Logs/LogFile.txt
```

When a user reports an error, ask for the correlation ID from the 500 response body. This lets you find all log lines for that exact request.

## Common Error Scenarios

### 401 Unauthorized

**Cause:** JWT token is missing, expired, or from the wrong IdentityServer issuer.

**Debug steps:**
1. Decode the JWT at [jwt.io](https://jwt.io) (do not use production tokens on public tools)
2. Check `exp` claim — is the token expired?
3. Check `iss` claim — does it match the configured IdentityServer URL?
4. Check `aud` claim — does it match the API's expected audience?
5. Enable `Microsoft.AspNetCore.Authentication` logging at `Debug` level

### 403 Forbidden from IP validation

**Cause:** The `IpValidateMiddleware` rejected the request IP.

**Debug steps:**
1. Check `ApplicationOptions:Common:IpSafeList` in config
2. The middleware logs the remote IP at `Debug` level — check the log
3. In local dev, set `IpSafeList` to empty string to disable

### 403 Forbidden from matter access check

**Cause:** `MatterRepository.IsMatterAuthorizedForIntegrator()` returned false.

**Debug steps:**
1. Check that the JWT's integrator key is registered for the matter
2. Check `GetMatterUserPermission()` — it calls a database UDF

### 400 Bad Request with `issues` array

**Cause:** `BadDataException` was thrown with validation details.

**Debug steps:**
1. Read the `issues` array in the response body — it contains field names and messages
2. Find the service method that threw the exception
3. Check the validation logic for the specific field

### 404 Not Found

**Cause:** `ResourceNotFoundException` thrown — the entity was not found in the database.

**Debug steps:**
1. Verify the ID passed in the request is correct
2. Check the tenant — if the ID exists in a different tenant's database, it will 404
3. Enable `DTA` debug logging to see the SQL query and its parameters

### 500 Internal Server Error

**Cause:** Unhandled exception. `ExceptionHandlingMiddleware` caught it.

**Debug steps:**
1. The 500 response body contains a `correlationId`
2. Search logs for that correlation ID
3. Look for `LogException` entries — they include the full stack trace

## Debugging Multi-Tenancy Issues

If data looks wrong or connections fail:

1. **Check the `OrgRef` claim in the JWT.** This is the tenant key. Decode the token and find `organisation_reference`.

2. **Verify the tenant is registered.** In multi-tenant mode, the `IMultiTenantService` implementation resolves connection strings from a registry. If `OrgRef` is not in the registry, the call fails.

3. **Check `Multitenant` config flag.** If `ApplicationOptions:Common:Multitenant = false`, `BaseMultiTenantService` is used, which ignores `OrgRef` and uses the fixed `ConnectionStrings:Database`.

4. **Enable `DTA` debug logging.** The `GetConnectionString()` call in `BaseRepository` logs the resolved connection (without the password) at Debug level.

## Debugging AutoMapper Failures

AutoMapper validation runs at startup. If the API fails to start with a mapping error:

```
AutoMapper.AutoMapperConfigurationException:
  Unmapped members were found for type XXX
```

**Fix:**
1. Find the `{Domain}Profile` for the type
2. Add the missing `.ForMember()` mapping
3. If the member should be ignored: `.ForMember(d => d.Field, opt => opt.Ignore())`

---

## Debugging Stored Procedure Failures

If a stored procedure call fails silently or returns wrong data:

1. Enable `DTA` debug logging
2. Find the SQL parameters in the log
3. Run the stored procedure directly in SSMS with those parameters
4. Check for NULL parameters — `CreateParameter()` handles nulls but stored procs may not

---

## Debugging Caching Issues

If data appears stale:

1. Check `ApplicationOptions:CachingService:Using` — if it's `1` (In-Memory), data is cached in the process
2. In development, restart the API to clear the in-memory cache
3. Some caches have TTL — `MatterService` caches document folder structure for 5 minutes

## Swagger for Manual Testing

Swagger UI is available at `https://localhost:7186/swagger` when running locally.

1. Click "Authorize" and enter: `Bearer <your-jwt-token>`
2. All endpoints will use that token for authentication
3. Swagger shows all request/response schemas including `SwaggerResponse` annotations
4. Use "Try it out" to test individual endpoints without needing Postman

---

## Checking Application Startup

If the API refuses to start, examine the console output carefully:

1. **AutoMapper errors** — profile misconfiguration
2. **DI registration errors** — missing `AddScoped<>` in `Program.cs`
3. **Connection string errors** — User Secrets not set
4. **Certificate errors** — run `dotnet dev-certs https --trust`
5. **Port conflict** — another process on port 7186
